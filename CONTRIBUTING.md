# Contributing to WP Membership Registration Plugin

## Requirements

- Docker and Docker Compose
- PHP 8.0+ (for running Composer/tools outside Docker)
- Composer

## Local Development

### Start the environment

```bash
docker compose up -d
```

WordPress is available at http://localhost:8080.
Mailpit (email testing UI) is available at http://localhost:8025.
The plugin is mounted directly into the container — file changes are reflected immediately without rebuilding.

### First-time setup

```bash
composer install
```

Composer's `post-install-cmd` automatically runs Strauss to namespace-prefix vendor libraries into `vendor-prefixed/`. If you add a new production dependency, re-run:

```bash
composer run prefix-namespaces
```

### Activate the plugin

In the WordPress admin (http://localhost:8080/wp-admin), go to Plugins and activate **WP Membership Registration**. Or via WP-CLI inside the container:

```bash
docker compose exec wordpress wp plugin activate wp-membership-registration --allow-root
```

## Running Tests

Unit tests use PHPUnit with Brain Monkey (no WordPress bootstrap required):

```bash
composer run phpunit
```

Tests live in `tests/Unit/`. New test classes must follow the Brain Monkey setUp/tearDown pattern established in `tests/Unit/Email/MailerTest.php`. The `tests/bootstrap.php` file registers the PSR-4 autoloader and global i18n stubs.

## Coding Standards

```bash
composer run phpcs
```

The plugin enforces WordPress Coding Standards (WPCS) via PHPCS. The `.phpcs.xml` config:
- Targets `src/` and `wp-membership-registration.php`
- Excludes PSR-4 filename sniffs (`WordPress.Files.FileName.*`) to allow `ClassName.php` files
- Requires proper escaping, sanitization, and nonce verification on all user input

All pull requests must pass `composer run phpcs` with zero errors.

## Project Structure

```
src/
├── Admin/          # Settings page, registrar, AJAX handlers
├── Email/          # Mailer class
├── Frontend/       # FormRenderer (HTML form rendering)
├── Pdf/            # PdfGenerator (TCPDF-based)
├── Shortcodes/     # MembershipFormShortcode
└── Util/           # FieldSchema helper

assets/
├── css/            # admin-settings.css, form.css
└── js/             # admin-settings.js, form.js

templates/
└── admin-settings-page.php

tests/
└── Unit/
    ├── Admin/
    ├── Email/
    ├── Frontend/
    └── Pdf/

vendor/             # Composer dependencies (not committed)
vendor-prefixed/    # Strauss-prefixed vendor files (not committed — generated by prefix-namespaces)
```

## Vendor Namespacing

Production dependencies (currently TCPDF) are namespace-prefixed by Strauss to avoid conflicts with other plugins. The Strauss configuration is in `composer.json` under `extra.strauss`. The `vendor-prefixed/` directory is generated and excluded from git. After `composer install`, Strauss runs automatically via `post-install-cmd`.

## Pull Request Checklist

- [ ] `composer run phpcs` passes with zero errors
- [ ] `composer run phpunit` passes with zero failures
- [ ] New PHP classes follow PSR-4 naming in `src/` under `WpMembershipRegistration\` namespace
- [ ] All user input is sanitized; all output is escaped (WPCS rules enforced)
- [ ] New AJAX handlers call `check_ajax_referer()` as the first line
- [ ] No files written to `wp-content/uploads/` — PDFs use `sys_get_temp_dir()` + `wp_delete_file()`
- [ ] New production dependencies added to `extra.strauss.packages` if they need prefixing
